# 顧客要件ヒアリングスキル - 実装版

このドキュメントは、顧客との対話を通じて要件定義書を生成するための実装ガイドです。

---

## 使用方法

### 実行コマンド（推奨）

以下のいずれかの方法でスキルを実行できます：

```bash
# オプション1: スキルとして登録している場合
/customer-requirements

# オプション2: 直接プロンプトとして実行
```

### 実行フロー

スキルを実行すると、以下の4フェーズを順番に進めます：

1. **ビジネス背景・目標** (5問)
2. **ユーザーペルソナ** (4問)
3. **機能要件** (3問)
4. **非機能要件** (3問)

---

## フェーズ 1: ビジネス背景・目標

### 質問 1-1
**プロジェクトの概要と名称を教えていただけますか？**

_例）「ECサイトリプレイス」「社内業務自動化ツール」など_

**回答のポイント：**
- プロジェクト名
- 簡潔な説明（1-2文）
- 関連するステークホルダー（主催部門など）

---

### 質問 1-2
**なぜこのプロジェクトが今、必要なのでしょうか？現在直面している課題や機会はありますか？**

**掘り下げ質問（不十分な場合）：**
- その課題が生じている背景は？
- どの程度の頻度/規模で影響が出ていますか？
- これまでどのような対策を取られていましたか？

**回答のポイント：**
- 具体的な課題（定量的なデータがあれば理想）
- 課題の深刻度
- 機会があれば記述

---

### 質問 1-3
**このプロジェクトで期待される効果やビジネス目標は何でしょうか？**

_例）「売上10%増加」「業務効率 50% 削減」「ユーザー満足度向上」など_

**掘り下げ質問（不十分な場合）：**
- その目標の測定方法は？（KPI）
- いつまでに達成したいですか？
- 成功とはどのような状態ですか？

**回答のポイント：**
- 明確で測定可能な目標
- 期限（目標達成時期）
- 成功指標（KPI）

---

### 質問 1-4
**このシステムのターゲット市場やユーザー層はどのようなものですか？**

_例）「20代～40代の都市部ユーザー」「B2B 企業の経営層」など_

**掘り下げ質問（不十分な場合）：**
- ユーザーの主な属性は？（年齢、職種、経験年数など）
- 推定ユーザー数は？
- 地域的な制限はありますか？
- グローバル展開の予定は？

**回答のポイント：**
- 具体的なユーザー属性
- 推定ユーザー規模
- 地域/国別の分布（あれば）

---

### 質問 1-5
**競合製品や既存ソリューションと比べて、貴社の差別化ポイントは何ですか？**

**掘り下げ質問（不十分な場合）：**
- 競合製品の課題は何か？
- 貴社の強み・独自性は何か？
- 価格戦略は？
- ブランド/信頼性でのアドバンテージは？

**回答のポイント：**
- 競合製品との主な違い
- 自社の強み（技術、コスト、サービスなど）
- 市場での位置付け

---

## フェーズ 2: ユーザーペルソナ

### 質問 2-1
**主なユーザーはどのような人物ですか？具体的に説明していただけますか？**

_例）「営業部長、35歳、15年の営業経験」など_

**掘り下げ質問（不十分な場合）：**
- その人物の職種は？
- 組織内での役割は？
- 日常的な業務内容は？
- その人物の課題や不満は？

**回答のポイント：**
- 職種
- 役職
- 経験年数
- 典型的な1日の仕事内容

---

### 質問 2-2
**そのユーザーの具体的なニーズや課題は何ですか？このシステムがどう役立つべきですか？**

**掘り下げ質問（不十分な場合）：**
- 現在、どのような課題に直面していますか？
- 解決したいことの優先順位は？
- 理想的な体験はどのようなものですか？

**回答のポイント：**
- ユーザーが抱える主な課題
- 課題の深刻度/発生頻度
- 理想的なソリューション像

---

### 質問 2-3
**ユーザーの技術リテラシーレベルはどのくらいですか？**

_初心者 / 中級者 / 上級者 / エキスパート から選択_

**掘り下げ質問（不十分な場合）：**
- 類似システムの利用経験はありますか？
- 今回が初めてのデジタルツール導入ですか？
- 技術サポートの必要性はどのくらい？
- トレーニング期間はどのくらい必要？

**回答のポイント：**
- テックリテラシーレベル
- 類似ツール経験の有無
- サポート/トレーニング必要性

---

### 質問 2-4
**ユーザーはどのような環境でこのシステムを使いますか？**

_例）「オフィスでPC」「外出先でスマートフォン」など_

**掘り下げ質問（不十分な場合）：**
- 主なデバイスは？（PC、タブレット、スマートフォン）
- オンライン/オフライン環境？
- ネットワーク環境の品質は？
- 1日のアクティブ時間は？

**回答のポイント：**
- 主なデバイス
- 使用場所（オフィス、外出先、自宅など）
- ネットワーク環境
- 1日あたりの平均使用時間

---

## フェーズ 3: 機能要件

### 質問 3-1
**サービス開始時に絶対に必要な機能は何ですか？（MVP）**

_まずはこれがないと始まらない、という機能をリストアップしてください。_

**掘り下げ質問（不十分な場合）：**
- 1番重要な機能は？
- その機能がないと、ユーザーは困りますか？
- 優先順位をつけると、TOP 3 は？
- 各機能の実装難易度は低い？高い？

**回答のポイント：**
- MVP に含める機能のリスト
- 各機能が必須である理由
- 機能の優先順位

**出力形式例：**
```
【MVP 機能】
1. ユーザー認証・ログイン機能（重要度：最高）
2. メイン機能A（重要度：最高）
3. 基本的なレポート機能（重要度：高）
```

---

### 質問 3-2
**MVP の後に追加したい機能や、「あったら嬉しい」機能はありますか？**

**掘り下げ質問（不十分な場合）：**
- その機能は必須ですか、それとも nice-to-have ですか？
- いつ頃実装したいですか？
- その機能によるユーザー価値は？

**回答のポイント：**
- MVP 後の拡張機能リスト
- 各機能の実装時期（v1.1、v2.0 など）
- 必須/オプション の分類

**出力形式例：**
```
【拡張機能】
- 高度なレポート機能（v1.1 で実装予定）
- API 連携機能（v2.0 で実装予定）
- AI による自動分類機能（将来検討）
```

---

### 質問 3-3
**これまで挙げられた機能を、優先度順に並べていただけますか？**

以下のマトリックスを参考に、「ビジネス価値」vs「実装難易度」で分類してください。

| | 低難易度 | 高難易度 |
|---|---------|---------|
| **高価値** | 最優先 | 次優先 |
| **低価値** | 後回し | 不要 |

**掘り下げ質問（不十分な場合）：**
- なぜその優先度ですか？
- スケジュール制約はありますか？
- 依存関係のある機能は？

**回答のポイント：**
- 優先度マトリックス（上記表を埋める）
- 各機能の実装順序
- スケジュール上の制約

---

## フェーズ 4: 非機能要件

### 質問 4-1
**システムのパフォーマンスについて、どのような要件がありますか？**

_例）「ページ読み込み3秒以内」「同時ユーザー 10,000 対応」など_

**掘り下げ質問（不十分な場合）：**
- 想定される同時アクセス数は？
- レスポンス時間はどのくらい必要ですか？
- ピーク時間帯はいつですか？
- スケーラビリティはどのくらい必要？

**回答のポイント：**
- 想定同時ユーザー数
- レスポンスタイム目標
- ピークトラフィック時の対応要件
- スケーラビリティの必要度

**出力形式例：**
```
【パフォーマンス要件】
- ページ読み込み時間：3秒以内
- 同時ユーザー対応：10,000名
- ピーク時対応：営業時間中（9:00-18:00）
```

---

### 質問 4-2
**セキュリティについて、どのような要件がありますか？**

_例）「個人情報の保護」「データ暗号化」「アクセス制御」など_

**掘り下げ質問（不十分な場合）：**
- 扱うデータの機密性レベルは？
- コンプライアンス要件（GDPR など）はありますか？
- 認証方式は？（ID/パスワード、SSO など）
- 監査ログの必要性は？

**回答のポイント：**
- 扱うデータの種類と機密性
- 適用法規/コンプライアンス
- 認証/認可方式
- 監査・ログ要件

**出力形式例：**
```
【セキュリティ要件】
- 個人情報保護：GDPR 準拠
- 認証：SSO（社内 Active Directory 連携）
- 通信暗号化：全て HTTPS
- 監査ログ：全操作を記録・保持（1年間）
```

---

### 質問 4-3
**システムの信頼性や保守性について、要件はありますか？**

_例）「可用性 99.9%」「24/7 監視」「自動バックアップ」など_

**掘り下げ質問（不十分な場合）：**
- 許容できなダウンタイムは？
- バックアップ戦略は？
- ログ/監視はどのくらい必要？
- 災害時の復旧目標（RTO/RPO）は？

**回答のポイント：**
- 可用性目標（SLA）
- バックアップ戦略
- 監視・ロギング要件
- 災害復旧要件（RTO/RPO）

**出力形式例：**
```
【信頼性・保守性要件】
- 可用性：99.9% (年間ダウンタイム 43分以内)
- バックアップ：日次自動バックアップ、復旧時間 < 1時間
- 監視：24/7 自動監視、アラート通知
- 災害復旧：RTO = 4時間、RPO = 1時間
```

## スキルの効果的な使用方法

### ベストプラクティス

1. **事前準備**
   - 顧客との面談を30分程度確保
   - 顧客側で意思決定者を揃える
   - 基本的なプロジェクト概要を事前に共有

2. **進行中**
   - 各フェーズの間で軽く要約する
   - わかりにくい回答には追加質問を実施
   - 時間配分に注意（各フェーズ 5-10分程度）

3. **完了後**
   - 生成された要件定義書をレビュー
   - 必要に応じて追加質問や修正を実施
   - 実装チームと要件の認識齟齬を確認

### よくある課題と対応

| 課題 | 対応方法 |
|------|---------|
| 顧客が抽象的な回答をする | 具体例を提示して掘り下げる |
| 機能要件が多すぎる | MVPを明確にして優先度付けを進める |
| 非機能要件の認識がない | なぜそれが必要か をビジネス面から説明する |
| 回答が一貫していない | 矛盾を指摘して整理する |

## このスキル後の対応

このスキルは **要件の初期収集** を目的としています。

次のステップとして、以下を推奨します：

1. **詳細仕様書の作成** → 要件定義書を基に、UI/UX、DB設計など詳細仕様を策定
2. **実装計画の作成** → 機能の優先順位、スプリント計画、リソース配分
3. **テスト計画の作成** → 受け入れ基準、テストケース設計
4. **その他の計画** → リスク管理、セキュリティ詳細設計、など
