# 顧客要件ヒアリングスキル - 使用ガイド

## 📋 スキル概要

このスキルは、顧客との対話を通じて新規プロジェクトの要件を体系的に収集し、実装チームが使用可能な構造化された要件定義書を自動生成します。

**所要時間**: 約 20～30分
**出力**: マークダウン形式の要件定義書

---

## 🚀 クイックスタート

### 使い方（3ステップ）

```
1. 本スキルを実行
   → セクション1: ビジネス背景・目標（5問）

2. 各質問に回答
   → セクション2: ユーザーペルソナ（4問）
   → セクション3: 機能要件（3問）
   → セクション4: 非機能要件（3問）

3. 要件定義書が自動生成される
   → マークダウン形式で提示されるので、コピペで活用可能
```

## 🎯 各フェーズの詳細

### フェーズ 1: ビジネス背景・目標（5問）

**目的**: プロジェクト全体の文脈を理解する

| 質問 | 目的 | 出力に含まれる内容 |
|------|------|-----------------|
| 1. プロジェクト概要 | プロジェクト名・規模を把握 | プロジェクト名、規模感、期間 |
| 2. 課題・機会 | なぜ今必要か を理解 | 現状の課題、背景、緊急性 |
| 3. ビジネス目標 | 成功定義を明確化 | 目標値（定量的）、KPI、期限 |
| 4. ターゲット市場 | ユーザー層を把握 | ユーザーセグメント、規模、地域 |
| 5. 競争優位性 | 市場での位置付けを明確化 | 差別化要素、競合優位性 |

**期待される出力**:
```
【プロジェクト概要】
- 名称: 社内営業管理システムのクラウド化
- 規模: 中規模（営業部門 50名を対象）
- 期間: 6ヶ月以内での実装

【ビジネス目標】
- 営業効率 30% 向上
- 顧客管理データの一元化
- 在宅勤務対応
```

---

### フェーズ 2: ユーザーペルソナ（4問）

**目的**: ユーザーを深く理解し、UI/UX 設計の基盤を作る

| 質問 | 目的 | 出力に含まれる内容 |
|------|------|-----------------|
| 1. ユーザー属性 | 誰がこのシステムを使うか | 職種、役職、経験年数、スキル |
| 2. ユーザーニーズ | どのような課題を解決するか | 具体的な業務課題、解決策 |
| 3. 技術レベル | どのレベルの UX が必要か | 技術リテラシー、トレーニング必要性 |
| 4. 使用環境 | どこでどう使うか | デバイス、場所、使用パターン |

**期待される出力**:
```
【主ペルソナ】
- 職種: 営業部員
- 年齢: 30～45歳
- 経験: 営業経験 5～10年
- 技術レベル: 中級（Office 程度の使用経験）

【ユーザーニーズ】
- 顧客情報を素早く参照したい
- 営業活動の記録を効率的に残したい
- 在宅勤務中も情報にアクセスしたい

【使用環境】
- デバイス: PC（主）、スマートフォン（補助）
- 場所: オフィス（60%）、顧客先（30%）、在宅（10%）
```

---

### フェーズ 3: 機能要件（3問）

**目的**: 実装すべき機能を優先度付けして明確化

| 質問 | 目的 | 出力に含まれる内容 |
|------|------|-----------------|
| 1. MVP 機能 | 最初に実装すべき機能を決定 | 必須機能のリスト、理由 |
| 2. 拡張機能 | 将来の展開方向を明確化 | 次フェーズ以降の機能 |
| 3. 優先度付け | 実装順序を決定 | 優先度マトリックス、スケジュール |

**期待される出力**:
```
【MVP 機能】
- ユーザー認証・ログイン
- 顧客情報管理（検索・閲覧・編集）
- 営業活動ログ（記録・参照）
- 基本的なレポート（顧客一覧、活動履歴）

【v1.1 以降の拡張機能】
- メール統合
- カレンダー連携
- AI による営業提案

【優先度】
| 機能 | 価値 | 難易度 | 優先度 |
|------|------|--------|--------|
| 認証 | 高 | 低 | 最優先 |
| 顧客管理 | 高 | 低 | 最優先 |
| 活動ログ | 高 | 中 | 次優先 |
| レポート | 中 | 中 | 次優先 |
| メール統合 | 中 | 高 | v1.1 |
```

---

### フェーズ 4: 非機能要件（3問）

**目的**: システムの性質上重要な要件（パフォーマンス、セキュリティなど）を明確化

| 質問 | 目的 | 出力に含まれる内容 |
|------|------|-----------------|
| 1. パフォーマンス | システムの速度・容量要件 | 同時ユーザー数、レスポンス時間 |
| 2. セキュリティ | 情報保護要件 | 認証方式、データ暗号化、コンプライアンス |
| 3. 信頼性・保守性 | 稼働継続性要件 | 可用性、バックアップ、災害復旧 |

**期待される出力**:
```
【パフォーマンス要件】
- 同時ユーザー: 50名
- ページロード時間: 3秒以内
- データ容量: 顧客データ 10,000件以上

【セキュリティ要件】
- 認証: AD（Active Directory）連携
- 通信: HTTPS 必須
- データ保護: 顧客情報は暗号化
- コンプライアンス: 個人情報保護方針準拠

【信頼性・保守性要件】
- 可用性: 営業時間中 99%
- バックアップ: 日次
- 復旧時間: 2時間以内
```

## 📊 出力される要件定義書の活用

### 出力フォーマット

生成される要件定義書は以下の構成です：

```
1. ビジネス背景・目標
   ├─ プロジェクト概要
   ├─ 課題・機会
   ├─ ビジネス目標
   ├─ ターゲット市場/ユーザー層
   └─ 競争優位性

2. ユーザーペルソナ
   ├─ ペルソナ概要
   ├─ ユーザーニーズ
   ├─ 技術レベル
   └─ 使用環境

3. 機能要件
   ├─ MVP 機能リスト
   ├─ 拡張機能リスト
   └─ 優先度マトリックス

4. 非機能要件
   ├─ パフォーマンス要件
   ├─ セキュリティ要件
   └─ 信頼性・保守性要件

5. 補足事項・制約条件
```

### 活用例

| 対象者 | 活用方法 |
|--------|---------|
| **プロダクトマネージャー** | 機能優先度付け、ロードマップ作成の基になる |
| **要件分析者** | 詳細仕様書作成の入力情報として活用 |
| **開発チーム** | 実装方針、システム構成の設計に活用 |
| **QA チーム** | 受け入れ基準、テストケース設計の基になる |
| **営業・カスタマーサクセス** | 顧客期待値の管理に活用 |

## ⚠️ よくある課題と対応

### Q1. 顧客が回答に時間がかかる場合

**対応方法**:
- 質問を 2～3 個に分割する
- 複数回のセッションに分ける
- 事前に簡単な事前ヒアリングを実施

---

### Q2. 回答が曖昧・抽象的な場合

**対応方法**:
- スキルが自動で「掘り下げ質問」を提示します
- より具体的な例を挙げるよう促す
- 数値化を心がける（定量的な回答を引き出す）

---

### Q3. 機能要件が多すぎてまとまらない場合

**対応方法**:
- MVP に絞り込む（「これがないと始まらない」とは何か）
- 優先度マトリックスで可視化
- フェーズ分けを明確にする（v1.0, v1.1, v2.0, など）

---

### Q4. 出力された要件定義書に不足がある場合

**対応方法**:
- 出力された要件定義書をレビュー
- 不足箇所を追加質問で補足
- 詳細仕様書は別途作成（このスキルの後に実施）
