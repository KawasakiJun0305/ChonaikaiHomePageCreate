# プロジェクト要件定義フォルダ

このフォルダは、プロジェクトの全体的な要件定義から詳細設計までのドキュメントを管理しています。

---

## 📋 ドキュメント構成

新規プロジェクト開始時には、以下の順序でドキュメントを作成・更新してください。

### 1️⃣ **顧客要求書** (`01_顧客要求.md`)

**目的:** 顧客が実現したいビジネス目標と要求を明確にする

**対象者:** プロジェクト全体

**作成時期:** プロジェクト初期段階

**主な内容:**
- 顧客情報とビジネス背景
- 課題・痛点・解決したいビジョン
- 機能・非機能要求（ハイレベル）
- スコープ・制約・スケジュール
- 成功基準・リスク

**確認すること:**
- [ ] 顧客の真のニーズが理解できているか
- [ ] スコープと納期が現実的か
- [ ] リスクが十分に把握されているか

---

### 2️⃣ **要件定義書** (`02_要件定義.md`)

**目的:** 顧客要求を実現可能な詳細要件に翻訳する

**対象者:** 設計・開発チーム、品質保証チーム

**作成時期:** 顧客要求確定後

**主な内容:**
- ユースケース・機能一覧
- 非機能要件（パフォーマンス、セキュリティ等）
- UIプロトタイプ・画面仕様
- API仕様・データベース仕様
- ビジネスルール・制約条件

**確認すること:**
- [ ] すべての顧客要求がカバーされているか
- [ ] 要件が開発可能か（技術的実現性）
- [ ] テスト可能な要件となっているか
- [ ] 曖昧さがないか

---

### 3️⃣ **業界リサーチ** (`03_業界リサーチ.md`)

**目的:** 市場や業界の状況を理解し、プロジェクトの位置づけを明確にする

**対象者:** 企画・営業・マネージメント、設計・開発チーム

**作成時期:** プロジェクト企画～要件定義段階

**主な内容:**
- 市場分析（市場規模、成長率、セグメント分析）
- 競合分析（競合企業の機能・強み・弱み）
- 規制・コンプライアンス（適用法規制）
- 技術トレンド（業界で注目されている技術）
- ベストプラクティス・成功事例

**活用シーン:**
- 機能の優先順位付けを行う際の根拠
- 技術スタック選定の判断材料
- リスク評価や対応戦略の検討
- 営業・マーケティング戦略の立案

---

### 4️⃣ **基本設計書** (`04_基本設計.md`)

**目的:** システムの全体構成・アーキテクチャを決定する

**対象者:** 設計・開発チーム、インフラチーム

**作成時期:** 要件定義完了後

**主な内容:**
- システムアーキテクチャ図（全体構成）
- モジュール構成・機能分割
- データフロー図
- インターフェース設計
- セキュリティ設計
- パフォーマンス・スケーラビリティ戦略
- デプロイメント戦略

**確認すること:**
- [ ] スケーラビリティ要件を満たすアーキテクチャか
- [ ] セキュリティ要件が設計に反映されているか
- [ ] 既存システムとの連携は適切か
- [ ] チームスキルで実装可能か

---

### 5️⃣ **詳細設計書** (`05_詳細設計.md`)

**目的:** 実装に必要なすべての詳細を定義する

**対象者:** 開発チーム、テストチーム

**作成時期:** 基本設計完了後

**主な内容:**
- テーブル定義（ER図、カラム仕様）
- API仕様（エンドポイント、リクエスト/レスポンス）
- クラス設計（クラス図、メソッド仕様）
- 画面設計（レイアウト、入力フィールド）
- プログラムフロー（ロジック図、状態遷移図）
- エラーハンドリング詳細
- テスト戦略・テストケース例
- ログ・監視設定

**確認すること:**
- [ ] 開発者が実装を開始できる程度の詳細度か
- [ ] 実装の揺らぎがないか（設計の曖昧さがないか）
- [ ] テスト可能な設計となっているか

## 📝 テンプレート使用時のチェックリスト

### プロジェクト開始時

- [ ] 顧客要求書を顧客と協力して完成させた
- [ ] プロジェクトスコープが明確になった
- [ ] 初期リスク評価を実施した

### 要件定義フェーズ

- [ ] 業界リサーチを実施し、市場・競合・技術トレンドを把握した
- [ ] 要件定義書で全機能・非機能要件をリストアップした
- [ ] ステークホルダーのレビュー・承認を得た

### 設計フェーズ

- [ ] 基本設計でアーキテクチャを決定し、技術的実現性を確認した
- [ ] 詳細設計で実装レベルの詳細を定義した
- [ ] 設計レビューで品質を確保した

### 開発フェーズ

- [ ] すべての詳細設計が実装に反映された
- [ ] テストケースが詳細設計に基づいて作成された

## 💡 テンプレート作成時のヒント

### 情報の充実度について

**テンプレート内の記号:**
- `[  ]` - 入力すべき項目
- `[例]` - 記入例
- `# 見出し` - セクション見出し

### 各ドキュメントで特に重要なポイント

| ドキュメント | 特に重要な点 |
|----------|----------|
| 顧客要求書 | 顧客の真のニーズを把握し、スコープを明確にすること |
| 要件定義書 | 要件の曖昧さを排除し、テスト可能にすること |
| 業界リサーチ | 競合分析と技術トレンドをプロジェクト戦略に反映させること |
| 基本設計書 | スケーラビリティとセキュリティを考慮したアーキテクチャを提案すること |
| 詳細設計書 | 実装チームが迷わない程度の具体性を提供すること |
